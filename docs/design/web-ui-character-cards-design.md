## Problem Statement

fiona-helper is a Quarkus-based MCP Server prompt template engine with zero frontend. All interaction happens through MCP Tools (create_session, insert_slot_value, assemble_prompt, submit_score, etc.). There is no visual way to:

- Browse and manage characters (name, baseDesign, personality)
- View pipelines and their slots
- Upload reference images for characters
- See scores and reviews for completed sessions
- Manage prompts in a visual interface

The goal is to add a web UI that makes the backend data tangible — starting with the most exciting feature: character cards.

## What Makes This Cool

The "whoa" moment: a grid of character cards with uploaded avatars, click-to-expand detail view with personality tags, and a star-rating overlay showing how well each character's prompts score. It turns a database table into something you'd want to browse.

## Constraints

- **Local only** — runs on developer machine, no auth, no deployment
- **Learning project** — prioritize learning value over production polish
- **Start with character cards** — other features (prompt management, scoring UI) come in follow-up iterations
- **Backend-first** — existing Quarkus services (PipelineAssemblerService, SessionScoringService) stay unchanged. For character CRUD, no service layer exists yet — REST resources will call `CharacterRepository` directly (idiomatic for simple Panache CRUD). A `CharacterService` can be extracted later if business logic grows.
- **No separate frontend server in production** — Quarkus serves built static files from `META-INF/resources`. Development uses Vite dev server on port 5173 with proxy to Quarkus on 8080.

## Premises

1. **Local-only, no auth** — The UI runs on your machine. No login, no multi-user, no deployment pipeline needed.
2. **Frontend served by Quarkus** — The UI is static files (HTML/JS/CSS) served from the same Quarkus app. No separate frontend server.
3. **Image storage = local filesystem** — Character avatars/references stored as files on disk, referenced by path. No S3/cloud storage.
4. **Backend API = REST, not MCP** — The UI calls REST endpoints (not MCP tools). New REST controllers wrap existing services.
5. **Character cards first, rest later** — The first milestone is a working character CRUD with cards. Prompt management, scoring, etc. come in follow-up iterations.

## Approaches Considered

### Approach A: Vanilla HTML/JS — "Ship today"
Pure HTML + CSS + vanilla JavaScript. No build step, no framework. Quarkus serves static files from `META-INF/resources`. REST endpoints return JSON, frontend calls `fetch()`.

- **Effort:** Small
- **Risk:** Low
- **Pros:** Zero dependencies, no build tooling to learn, works immediately
- **Cons:** Manual DOM manipulation gets tedious, no component reuse, harder to scale to more features
- **Reuses:** Existing Quarkus setup, existing JPA entities/services

### Approach B: Vue 3 SPA — "The real deal" (RECOMMENDED)
Vue 3 + Vite. Component-based UI. Build step produces a bundle dropped into Quarkus's static resources. Clean separation between frontend and backend API.

- **Effort:** Medium
- **Risk:** Low
- **Pros:** Component system perfect for card grids, reactive data binding, great dev experience with hot reload, transferable skill
- **Cons:** Need to learn Vue/Vite basics, build pipeline adds complexity
- **Reuses:** Same backend services, REST endpoints needed regardless

### Approach C: Quarkus Qute + HTMX — "The hot take"
Server-side rendering with Qute templates. HTMX handles interactions by fetching HTML fragments from the server and swapping DOM. Almost zero client-side JavaScript.

- **Effort:** Small-Medium
- **Risk:** Medium
- **Pros:** No JS framework to learn, backend-driven UI, very fast for simple CRUD
- **Cons:** HTMX is less common knowledge, UI interactivity feels "page-refresh-ish," character card animations/grid layouts harder
- **Reuses:** Qute is a Quarkus-first template engine

## Recommended Approach

**Vue 3 SPA (Approach B).**

Rationale: You're in learning mode — Vue's component model is the best way to learn modern frontend architecture. Character cards with click-to-expand, image upload previews, and reactive star ratings are exactly what Vue excels at. The build step is a one-time setup, and Vite's dev server with hot reload makes iteration fast. The skill is transferable to any modern web project.

### REST API Design

All write endpoints (`POST`, `PUT`, `DELETE`) need `@Transactional` because Panache `persist()` / `delete()` require a transaction.

```
GET    /api/characters              → List<CharacterResponse>
GET    /api/characters/{id}         → CharacterResponse (404 if not found)
POST   /api/characters              → CharacterResponse (body: {name, baseDesign, personality})
                                     400 if name is empty/null
PUT    /api/characters/{id}         → CharacterResponse (body: {name, baseDesign, personality})
                                     404 if not found
DELETE /api/characters/{id}         → 204 No Content
                                     (orphaned avatar files on disk are NOT cleaned up — acceptable for local tool)

POST   /api/characters/{id}/avatar  → 200 OK (multipart: file)
                                     400 if not image/* or > 5MB
GET    /api/images/{characterId}/{filename} → image file (raw bytes)
                                     400 if filename contains path traversal
                                     404 if file not found
```

**CharacterResponse:**
```java
public record CharacterResponse(
    UUID id,
    String name,
    String baseDesign,
    String personality,
    String avatarUrl   // e.g., "/api/images/{characterId}/{filename}" or null
) {}
```

### File Upload Flow

1. User selects file in `<input type="file" accept="image/*">`
2. Vue validates file type (starts with `image/`) and size (< 5MB client-side hint)
3. Vue uploads via `FormData` + `fetch()` POST to `/api/characters/{id}/avatar`
4. Backend validates: MIME type starts with `image/`, file size < 5MB
5. Backend saves to `uploads/characters/{characterId}/{timestamp}_{filename}`
6. Backend updates `CharacterEntity.avatarPath` inside `@Transactional`
7. If DB update fails, transaction rolls back but file remains orphaned on disk. For a local tool, this is acceptable.
8. Frontend refreshes card to show new avatar URL

### Image Serving

Quarkus serves uploaded images from `/api/images/*` via a custom resource that reads from the filesystem:
```java
@GET
@Path("/images/{characterId}/{filename}")
@Produces("image/*")
public Response getImage(@PathParam UUID characterId, @PathParam String filename) {
    // Validate filename — reject path traversal attempts
    if (filename.contains("..") || filename.contains("/") || filename.contains("\\")) {
        return Response.status(400).entity("Invalid filename").build();
    }
    Path baseDir = Path.of("uploads/characters").toAbsolutePath().normalize();
    Path filePath = baseDir.resolve(characterId.toString()).resolve(filename).normalize();
    // Ensure resolved path stays within baseDir (path traversal guard)
    if (!filePath.startsWith(baseDir)) {
        return Response.status(403).build();
    }
    if (!Files.exists(filePath)) {
        return Response.status(404).build();
    }
    String contentType = Files.probeContentType(filePath);
    return Response.ok(filePath.toFile())
        .header("Content-Type", contentType != null ? contentType : "application/octet-stream")
        .build();
}
```

### Key UX Decisions

- **Card grid:** CSS Grid, 3-4 columns on desktop, 1 on mobile. Cards have hover lift effect.
- **Avatar:** Circular crop, fallback to initials if no avatar.
- **Personality tags:** Split `personality` by commas into colored pills.
- **Create flow:** Floating "+" button opens modal with form.
- **Edit flow:** Inline edit or modal — start with modal for simplicity.
- **Delete:** Confirm dialog. Soft-delete not needed (local tool, data is yours).

## Open Questions

1. Should personality be stored as comma-separated tags or a separate `character_tag` table? (Start with comma-separated; migrate later if needed.)
2. Should images be base64-inlined in the API response for small avatars, or always served as separate HTTP requests? (Separate requests — simpler, cacheable.)
3. Vue router (SPA routing) vs. simple component switching? (Start without vue-router; add when pipeline list and score pages arrive.)
4. Should scoring UI reuse the existing `SessionScoringService` or wrap `ScoringTools`? (Wrap the service directly — REST resources should not depend on MCP tools.)
